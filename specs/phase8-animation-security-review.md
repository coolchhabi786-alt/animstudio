# Phase 8 â€” Animation Studio: Security Review (OWASP Top 10)

## A01: Broken Access Control
- **Status**: PASS
- **Evidence**: `AnimationController` applies `[Authorize(Policy = "RequireTeamMember")]` at the class level. All four endpoints (estimate, approve, list clips, signed URL) inherit this policy.
- **Finding**: Cross-episode/cross-team access to clips is defended at two layers: (1) the `RequireTeamMember` policy validates the caller's team membership for the episode; (2) `GetAnimationClipSignedUrlHandler` explicitly rejects `clip.EpisodeId != query.EpisodeId` as `NOT_FOUND`, preventing enumeration of clipIds from other episodes.
- **Mitigation**: None needed.

## A02: Cryptographic Failures
- **Status**: PASS
- **Evidence**: Clip URLs are short-lived (60-second) signed SAS tokens generated by `BlobClipUrlSigner`. No cost data, approver identity, or clip URLs are logged.
- **Finding**: Dev implementation of `BlobClipUrlSigner` stamps an `?se=<expires>` query parameter rather than a cryptographic SAS signature. This is acceptable for local/dev only â€” production must use `BlobClient.GenerateSasUri` with a user delegation key.
- **Mitigation**: Tracked as a non-goal for Phase 8; to be replaced in Phase 9 (Post-Production / CDN hardening).

## A03: Injection
- **Status**: PASS
- **Evidence**: All DB access uses EF Core parameterised queries. `ApproveAnimationCommand.Backend` is a strongly-typed enum validated by FluentValidation (`IsInEnum()`); `EpisodeId` and `ApprovedByUserId` are `Guid` (`NotEmpty()`). The worker-job payload is serialised with `System.Text.Json`, not string concatenation.
- **Mitigation**: None needed.

## A04: Insecure Design
- **Status**: PASS
- **Evidence**: Approval is gated by a mandatory cost-estimate review on the client and an explicit POST body (`ApproveAnimationRequest.Backend`). `HasActiveJobAsync` prevents duplicate spend from double-submit: a second approval returns `409 ANIMATION_ALREADY_ACTIVE` instead of queuing another render. Rate caps default to `$0.056`/shot (Kling) and `$0`/shot (Local) via `Animation:Rates:*` configuration; rate overrides require Key Vault access.
- **Mitigation**: None needed.

## A05: Security Misconfiguration
- **Status**: PASS
- **Evidence**: No new secrets added. `Animation:Rates:*` values are non-sensitive pricing constants. `AzureBlob:ClipsCdnBase` is a public CDN hostname.
- **Mitigation**: None needed.

## A07: Authentication Failures
- **Status**: PASS
- **Evidence**: `AnimationController.ApproveAnimation` extracts the approver identity via `User.FindFirstValue(ClaimTypes.NameIdentifier)` â€” derived from the validated JWT, not from user input. Missing / unparseable claims reject with `401 UNAUTHORIZED` before the handler runs.
- **Mitigation**: None needed.

## A08: Software & Data Integrity
- **Status**: PASS
- **Evidence**: `AnimationJob` and `AnimationClip` both use EF Core `IsRowVersion` concurrency. State transitions are enforced in domain methods (`MarkRunning` requires `Approved`; `MarkCompleted` rejects terminal-state doubles). `AddDomainEvent` is the only way to emit `AnimationClipReadyEvent`, so a direct DB edit cannot forge a SignalR broadcast.
- **Mitigation**: None needed.

## A09: Security Logging & Monitoring
- **Status**: PASS
- **Evidence**: Approval event emits `AnimationJobApprovedEvent` containing `episodeId`, `backend`, `estimatedCostUsd`, and `approvedByUserId` â€” all audit-appropriate, no sensitive data. Completion emits `AnimationJobCompletedEvent` with the terminal status and actual cost. No clip URLs, SAS tokens, or user claim values are logged.
- **Mitigation**: None needed.

## A10: Server-Side Request Forgery (SSRF)
- **Status**: PASS
- **Evidence**: `BlobClipUrlSigner` only concatenates the configured `AzureBlob:ClipsCdnBase` onto the stored `clip.ClipUrl` (a blob path produced by the worker, not user input). The API never fetches clip URLs server-side â€” clients pull directly from the signed CDN URL.
- **Mitigation**: None needed.

## Phase-specific: Cost / Budget Abuse
- **Status**: PASS
- **Evidence**: Approval is idempotent-by-rejection (`ANIMATION_ALREADY_ACTIVE`, 409) â€” replaying the POST cannot double-spend. `EstimatedCostUsd` is frozen into the `AnimationJob` row at approval time so a later rate change cannot retroactively inflate or deflate an approved run. The `authenticated` rate-limit policy (100 req/min/user) bounds brute-force attempts against the approval endpoint.
